Everything you need to clone, install, and run WinSTT locally for development.

## Prerequisites

Before you start, make sure you have these installed:

| Tool | Purpose | Install |
|------|---------|---------|
| [Git](https://git-scm.com/) | Version control, cloning dependencies | [Download](https://git-scm.com/download/win) |
| [uv](https://docs.astral.sh/uv/) | Python package manager (installs Python too) | See step 1 below |
| [Bun](https://bun.sh/) | JavaScript runtime + package manager | [Download](https://bun.sh/) |

**Optional but recommended:**

| Tool | Purpose |
|------|---------|
| NVIDIA GPU + [CUDA 12.4](https://developer.nvidia.com/cuda-toolkit) | GPU-accelerated inference via the `gpu` extra (`onnxruntime-gpu`) |
| [Ollama](https://ollama.com) | Local LLM-powered text enhancement / custom transforms (optional) |
| [OpenRouter](https://openrouter.ai) API key | Cloud LLM alternative to Ollama (optional) |

<Callout title="No Python install needed" type="info">
  uv downloads and manages Python for you. You do **not** need a system-wide Python installation.
</Callout>

## One-Click Setup

Run the batch script from the repo root — it handles everything automatically:

```bat
setup-dev.bat
```

The script will:
1. Install uv and Bun if missing
2. Clone the `onnx-asr` local dependency
3. Install Python 3.11 + all server packages (ONNX runtime via the `cpu` extra)
4. Install all frontend packages

If you prefer to do it manually, follow the steps below.

## Manual Setup

<Steps>

<Step>
### Install uv

Open PowerShell and run:

```powershell
irm https://astral.sh/uv/install.ps1 | iex
```

Then add `C:\Users\<your-username>\.local\bin` to your system PATH.

Restart your terminal and verify:

```bash
uv --version
```
</Step>

<Step>
### Install server dependencies

The server depends on `onnx-asr` via a [WinSTT-side fork](https://github.com/winstt/onnx-asr) pinned to a specific commit in `server/pyproject.toml` (`[tool.uv.sources]`). `uv sync` fetches it automatically — no separate clone step is needed.

The transcription stack is **ONNX-only** (no PyTorch in core deps). You must
pick exactly one ONNX runtime extra — `cpu`, `directml`, `openvino`, and `gpu`
are mutually exclusive (they share the `onnxruntime/` package directory):

```bash
cd server
uv sync --extra cpu          # CPU-only — smallest install
uv sync --extra directml     # DirectX 12 — works on any modern Windows GPU (recommended default)
uv sync --extra openvino     # Intel ARC dGPU or recent Iris Xe / Arc iGPU
uv sync --extra gpu          # NVIDIA CUDA — ~2 GB (Linux NVIDIA build path)
```

Each GPU extra auto-falls back to CPU at runtime if its EP isn't viable on the
host. The runtime priority order lives in
`server/src/recorder/infrastructure/device.py::_AUTO_PRIORITY`.

Optionally add feature extras:

```bash
uv sync --extra directml --extra tts --extra sentence-classifier
```

- `tts` — Kokoro-82M text-to-speech (`kokoro-onnx`, torch-free)
- `sentence-classifier` — optional Smart Endpoint / DistilBERT classifier (the only extra that pulls in `torch` + `transformers`; fails soft if absent)

This will:
- Download Python 3.11 if not already installed
- Create a `.venv` in `server/`
- Install all selected packages

<Callout title="An extra is required" type="warn">
  Plain `uv sync` with no extra installs no ONNX runtime variant. Always
  pass `--extra cpu`, `--extra directml`, `--extra openvino`, or `--extra gpu`.
</Callout>

Verify the install:

```bash
uv run stt-server --help
```
</Step>

<Step>
### Install frontend dependencies

```bash
cd frontend
bun install
```

Verify the Electron build compiles:

```bash
bun electron:compile
```
</Step>

</Steps>

## Running the App

You need **two terminals** — one for the server, one for the frontend.

<Tabs items={['Server', 'Frontend']}>
<Tab value="Server">
```bash
cd server
uv run stt-server
```

The server starts two WebSocket channels:
- **Control** (JSON commands): `ws://localhost:8011`
- **Data** (binary audio): `ws://localhost:8012`

Common flags:

| Flag | Description |
|------|-------------|
| `-m tiny.en` | Use a smaller model for faster startup |
| `--device cpu` | Force CPU inference (no CUDA) |
| `-c 8011 -d 8012` | Set custom WebSocket ports |
| `--no-enable_realtime_transcription` | Disable live preview |
| `-D` | Enable debug logging |

</Tab>
<Tab value="Frontend">
```bash
cd frontend
bun electron:dev
```

This compiles the Electron main/preload scripts and launches the app. The renderer connects to the STT server via IPC relay (the Electron main process holds the WebSocket connection).

</Tab>
</Tabs>

## Dev Commands Reference

### Server (`server/`)

| Command | Description |
|---------|-------------|
| `uv run stt-server` | Start the STT server |
| `uv run pytest` | Run all tests |

| `uv run ruff check . --fix` | Lint with auto-fix |

## Packaging

WinSTT ships **three portable installers per release**. All three wrap the same Electron
app; only the bundled `stt-server.exe` differs:

| Installer | Size | Contents |
|-----------|------|----------|
| `WinSTT-CPU-Portable-<version>.exe` | ~150 MB | `onnxruntime` (CPU-only) |
| `WinSTT-Portable-<version>.exe` (DirectML, default) | ~200 MB | `onnxruntime-directml` — DirectX 12 GPU acceleration on AMD / Intel / NVIDIA; auto-falls-back to CPU on hosts without a D3D12-capable GPU |
| `WinSTT-OpenVINO-Portable-<version>.exe` | ~250 MB | `onnxruntime-openvino` — Intel ARC dGPU / Iris Xe / Arc iGPU; auto-picked ahead of DirectML on Intel silicon when the EP is registered |

The legacy CUDA-bundle build (`onnxruntime-gpu` + 8 NVIDIA cu12 wheels, ~2 GB) was retired for Windows because DirectML is faster on our workload, more consistent in tail latency, and ~10× smaller. CUDA EP detection is preserved in `server/src/recorder/infrastructure/device.py` for the eventual Linux NVIDIA build.

Build the server executables, then the installers. Both commands run from the **repo root**. Packaging configs and intermediate PyInstaller output live under `packaging/`:

```
packaging/
├── electron-builder.yml
├── electron-builder.cpu.yml
├── electron-builder.directml.yml
├── electron-builder.openvino.yml
└── stt-server-dist/
    ├── cpu/
    ├── directml/
    └── openvino/
```

The final installer lands at `<repo>/dist/`.

<Tabs items={['CPU', 'DirectML (default GPU)']}>
<Tab value="CPU">
```bash
pwsh server/packaging/build.ps1 -Flavor cpu        # -> packaging/stt-server-dist/cpu/
bun run electron:build:cpu                         # -> dist/
```
</Tab>
<Tab value="DirectML (default GPU)">
```bash
pwsh server/packaging/build.ps1 -Flavor directml   # -> packaging/stt-server-dist/directml/
bun run electron:build:directml                    # -> dist/
```
</Tab>
</Tabs>

Tagging a release runs the CPU and DirectML builds as a CI matrix and publishes both
installers to the same GitHub Release ([github.com/winstt/WinSTT](https://github.com/winstt/WinSTT)):

```bash
git tag v0.X.0
git push --tags
```

## Troubleshooting

<Accordions>
<Accordion title="uv sync installs no ONNX runtime / 'no module named onnxruntime'">
Plain `uv sync` with no extra installs neither ONNX runtime variant. Pass exactly one of `--extra cpu` or `--extra gpu` (they conflict). Add `--extra tts` and/or `--extra sentence-classifier` for the optional TTS and Smart Endpoint features.
</Accordion>

<Accordion title="stt-server starts but transcription is slow">
Check that the GPU runtime is being used. Run `uv run stt-server -D` and look for the device info in the logs. If it says `cpu`, install the server with `uv sync --extra gpu` (and the [CUDA Toolkit 12.4](https://developer.nvidia.com/cuda-toolkit)), or pass `--device cpu` and use a smaller model like `-m tiny.en`. The GPU runtime auto-falls back to CPU if CUDA isn't viable.
</Accordion>

<Accordion title="Electron app shows a blank window">
Make sure the STT server is running before launching the Electron app. The frontend expects the server on ports 8011/8012 by default.
</Accordion>

<Accordion title="'onnx-asr' not found during uv sync">
`onnx-asr` is fetched directly from `github.com/winstt/onnx-asr` (pinned commit in `server/pyproject.toml`); no local checkout is needed. If `uv sync` fails on the git fetch, check your network / GitHub access and rerun.
</Accordion>

<Accordion title="bun install fails on uiohook-napi">
The prebuilt NAPI binary that ships with `uiohook-napi` is loaded as-is — no rebuild is forced, so no MSVC is required for dev. If you've modified the native helpers (`frontend/electron/native/src/winstt-*.c`) and want to rebuild them, that step does need Visual Studio Build Tools — run `bun --cwd frontend run native:build` from a Developer Command Prompt.
</Accordion>
</Accordions>
