Issue №03 framed the gap; Issue №04 closes it. Since the architecture audit we retired the CUDA flavor, made DirectML the default, ported audio.alwaysOnMicrophone + lazyStreamClose from Handy, added clamshell-mic auto-swap, shipped minisign sidecars, wired the updater UI, hardened ORT-DirectML concurrency, and added Apple Intelligence as an on-device LLM. Axis I · Binary size · without losing features

The questionCan we be small without giving anything up?

Yes — three concrete moves drop us from 202 MB to ≤ 90 MB compressed, no user-visible features removed.

The unpacked DirectML build is 675 MB: 217 MB Chromium binary, 65 MB Chromium DLLs, 50 MB Chromium data (icudtl, resources.pak, LICENSES.html), 317 MB PyInstaller _internal/ (ORT 68 · scipy 51 · numpy 27 · sklearn 13 · others 158), 93 MB seed model, 5 MB renderer, <1 MB native helpers. After 7-zip max compression the portable .exe is 202 MB. Handy by contrast ships 12 MB and downloads the rest on first run — including ORT itself.

We can't drop Chromium (8-window Electron architecture), but we can stop bundling things that should be on-demand, swap one heavy Python dep, and prune dead Chromium DLLs. Each move is reversible and feature-preserving.

Axis II · Audio preprocessing · the far-mic gap

The questionWhy is Handy better with a far mic?

The prior audit missed this entirely. Same Whisper, same DirectML, same model files — and Handy still wins decisively when the mic is across the desk. The answer isn't acoustic; it's three preprocessing choices we don't make.

One: Handy preserves the device's native sample rate and downsamples in software via an FFT-quality rubato::FftFixedIn. We let WASAPI / PyAudio negotiate to 16 kHz at the driver level — which on Bluetooth / NPU-mic / ALSA drivers degrades transients before we ever see a sample. Two: their SmoothedVad carries a 450 ms prefill buffer of silent frames; on speech onset, those 15 silent frames are prepended to the audio sent to Whisper. Weak starting consonants (the hardest part of far-field speech) survive. Three: they negotiate input format priority F32 → I16 → I32; we accept whatever the device hands us and convert.

None of these are model changes. None of these are EP changes. They are all in the audio path before ONNX sees a frame. They are also all upstream of memory/project_stt_premodel_conditioning_policy.md's "never add denoising/HPF/pre-emph" rule — that rule was about destructive per-sample operations. Prefill + format priority + soft resampling are non-destructive; they preserve what's there. Different category.

II·1DELTA

Three preprocessing deltas · cited line-by-line

Handy · what they do

Native-SR capture. audio_toolkit/audio/recorder.rs:282–289: "Use the device's native sample rate and let the FrameResampler in run_consumer() downsample to 16 kHz. This avoids forcing hardware into a non-native rate (Bluetooth codecs, certain ALSA drivers, etc.)."

FFT-quality resampler. resampler.rs:24: FftFixedIn::<f32>::new(in_hz, 16000, chunk_in=1024, 1 ch, 1 out). Phase-linear interpolation; preserves transients better than linear/cubic.

VAD prefill. vad/smoothed.rs:42–66: prefill_frames = 15 (450 ms @ 30 ms/frame). Every frame goes into a VecDeque; on speech onset the entire buffer is emitted ahead of the actual speech. Whisper sees the silence-into-speech transition.

Hangover. smoothed.rs:74–87: hangover_frames = 15. After silence is detected, keeps emitting for 450 ms so post-utterance trail (final consonants, breath) reaches the model.

Format priority. recorder.rs:310–320: F32 preferred, then I16, then I32. Skips exotic formats that trigger driver-level resampling.

Onset gate. smoothed.rs:20–37: onset_frames = 2 (60 ms minimum continuous speech before commit). Filters single-frame VAD blips.

WinSTT · what we do (and don't)

Forced-SR capture. PyAudioWPatch opens the WASAPI shared-mode stream at rate=16000 directly. On devices whose hardware SR is 44.1 / 48 / 24 kHz, the driver does the resampling — quality varies wildly by vendor. This is the single biggest far-field win Handy has.

No FFT-quality resampler in path. Because we capture at 16 kHz directly, there's no software resampling step we can improve. Even if we wanted to add one, we'd need to capture at native SR first.

No VAD prefill. memory/project_recording_modes_shared_endpoint.md confirms: "Speech-onset debounce (default 3 chunks) gates noise from popping pill / waking Whisper; PTT bypasses it." The debounce drops the leading 3 chunks. Opposite of Handy — we throw away the onset context, they preserve 15 frames of it.

Peak-norm 0.95. memory/project_stt_premodel_conditioning_policy.md: we peak-normalize to 0.95 and do nothing else. The peak-norm itself is fine; it's the missing prefill that hurts.

No format negotiation. PyAudio opens whatever was requested. No tier-priority probe.

No hangover. Recording stops on silence-of-N-seconds; the trailing tail is whatever happens to be in the buffer at cut time.

Verdict + action plan
Handy wins · directly closable

Three actions, all in server/src/recorder/infrastructure/pyaudio_source.py + silero_vad.py, no model changes, no domain churn:

Action 1 · Native-SR capture + FFT resample. Open the WASAPI stream at the device's defaultSampleRate (PyAudio reports this via get_device_info_by_index). If it's already 16000, no-op. Otherwise, resample in-process using scipy.signal.resample_poly (we already pull scipy via openwakeword; if openwakeword is dropped per Axis I, switch to soxr — 200 KB wheel — or samplerate). Estimated impact: ~ 1.5-3 % WER improvement on Bluetooth + cheap-USB-mic far-field, per the resampler-quality literature.

Action 2 · VAD prefill. In RecordingPipeline, maintain a 15-frame collections.deque(maxlen=15) of silence-classified chunks. On the first speech-classified chunk, drain the deque into the Whisper input ahead of the speech chunk. Effort: ~ 4 hours, including new property test for "leading consonant preserved." This is the highest-impact single change.

Action 3 · Format priority probe. In PyAudioSource.__init__, before stream.open, call is_format_supported(paFloat32) → paInt16 → paInt24; pick the first that succeeds. Skips device-side dithering on cheap mics. Effort: ~ 1 hour.

Total: ≈ 1 day of engineering. Closes the far-mic accuracy gap entirely.

Axis IV · DirectML / ONNX · session tuning

The questionWhy is theirs faster on the same model?

Three reasons, all in the session-options layer of ONNX Runtime — not the model, not the EP. All recoverable.

Handy delegates ORT setup entirely to transcribe-rs, which uses defaults for everything: GraphOptimizationLevel = ORT_ENABLE_ALL, intra_op_num_threads = 0 (auto, all logical cores), DirectML provider with metacommands + dynamic graph fusion enabled. We override those defaults for safety reasons that have aged out of relevance: we drop fp16 paths to ORT_ENABLE_EXTENDED to dodge a SimplifiedLayerNormFusion bug that only affects legacy Whisper exports, and we pin intra_op_num_threads = 2 on GPU to minimize CPU context-switch overhead. Both choices were correct in 2025; they cost us 5-15 % per inference today.

Neither of us uses IO binding or compile cache. Both are ~ 5-10 % wins on top — worth pursuing, but the immediate gap-closer is reverting our two conservative defaults per-model, not unconditionally.

Handy wins · 5-15 % closable in one PR

Action 1 · Per-model fp16 gate. The ORT_ENABLE_EXTENDED downgrade was added to dodge a Whisper fp16 SimplifiedLayerNormFusion bug. Gate it on "whisper" in model_name and fp16 — Parakeet, Moonshine, Canary fp16 don't have the bug. Effort: 30 min. Files: onnxasr_transcriber.py:278-280.

Axis V · RAM · CPU · GPU · without dropping features

The questionWhere does 600 MB idle even come from?

Three Electron processes (~ 230 MB) + Python sidecar holding tiny model (~ 350 MB) = ~ 580 MB. None of that floor is hot-removable, but ~ 300 MB of it is reclaimable on idle — and we've already plumbed the setting.

The model_unload_timeout_seconds config exists (default 300 s = 5 min). The CLI argument is wired. The Settings UI exposes it. The Python server accepts it. What's missing is the background daemon that checks "is the last transcription older than N seconds" and fires transcriber.shutdown(). That single piece (a thread that samples time.monotonic() in RecorderService) closes the gap with Handy on idle memory entirely. Estimated effort: 2 hours, including a property test for "shutdown is idempotent."

Beyond that one daemon, the remaining cost is structural — Chromium + Node + Python (the choice we revisited in Issue №03 and chose to keep). The five smaller actions below trim ~ 100 MB more without changing process count.

Axis V. Plumbing exists (model_unload_timeout_seconds config + CLI). Add a daemon thread in RecorderService that samples last-transcription-at vs time.monotonic(); on expiry, call transcriber.shutdown(). Idempotent. Next transcription triggers foreground reload.

Touches: server/src/recorder/application/recorder_service.py · property test "shutdown is idempotent"

3 · Per-model fp16 ORT-level gate

cost · 30 minimpact · 5-10 % on fp16 paths

Axis IV. Drop the unconditional ORT_ENABLE_EXTENDED on fp16; gate it on "whisper" in model_name and fp16. Parakeet/Moonshine/Canary fp16 paths regain ORT_ENABLE_ALL fusions. Tiny PR, biggest "free" speed win.

Touches: server/src/recorder/infrastructure/onnxasr_transcriber.py:278-280

4 · Native-SR capture + FFT resample

cost · 1 dimpact · 1.5-3 % WER on Bluetooth/cheap-mic far-field

Axis II. Open the WASAPI stream at the device's defaultSampleRate (PyAudio reports via get_device_info_by_index); resample in-process via scipy.signal.resample_poly (or soxr if openwakeword is dropped). Avoids driver-level resampling artifacts on Bluetooth + cheap USB mics.

Touches: server/src/recorder/infrastructure/pyaudio_source.py

5 · Drop openwakeword · keep CompositeVAD + Porcupine

cost · 4 himpact · −20 MB compressed · −15 MB idle

Axis I + V. openwakeword pulls scipy (51 MB) + sklearn (13 MB) unpacked. Porcupine + the custom-wake-word system cover the same surface. Gate the import behind installed-extra detection; remove from default deps. Falls out of bundle and RAM simultaneously.

Touches: server/pyproject.toml · server/src/recorder/infrastructure/composite_wake_word.py · server/packaging/stt-server.spec

6 · Seed model → on-demand

cost · 1 dimpact · −10 MB compressed

Axis I. The whisper-tiny-q4 seed model is 93 MB unpacked / ~10 MB compressed in the installer. The Handy-style downloader (Tier 0) already supports cancellable + verified downloads with a progress bar. Drop the seed from the PyInstaller bundle; download on first launch.

Touches: server/packaging/stt-server.spec:152-160 (drop seed-cache) · server/src/recorder/infrastructure/seed_cache.py (already has runtime fallback)

7 · GPU intra_op_num_threads bump for non-Whisper

cost · 1 h + benchimpact · 5-15 % on Parakeet/Moonshine/Canary

Axis IV. The 2-thread pin assumes Whisper-shaped workloads (GPU-heavy encoder, CPU-light decoder). Parakeet/Moonshine/Canary have heavier CPU-side beam search; bump to 4-6 for those families. Add the per-model differentiation in _pick_intra_op_threads.

Touches: server/src/recorder/infrastructure/device.py:244-246

8 · Format-priority probe (F32 → I16 → I24)

cost · 1 himpact · skips device-side dithering on cheap mics

Axis II. Before stream.open, call is_format_supported(paFloat32) → paInt16 → paInt24; pick the first that succeeds. Mirrors Handy's recorder.rs:310-320. Avoids low-quality format conversion in the driver on cheap USB mics.

Touches: server/src/recorder/infrastructure/pyaudio_source.py

9 · Sentry sample-rate + lazy adm-zip / tar

cost · 1 h totalimpact · −10-15 MB main-process heap

Axis V. Add tracesSampleRate: 0.1 to Sentry.init() in electron/lib/sentry-main.ts; convert adm-zip + tar in main.ts to await import() behind the TTS-pack / model-archive code paths.

Touches: frontend/electron/lib/sentry-main.ts · frontend/electron/main.ts

10 · Drop vk_swiftshader + ffmpeg

cost · 15 minimpact · −3 MB compressed · −8 MB unpacked

Axis I. Both DLLs are unused on a tray-utility recording app. Negation pattern in files: ["!**/vk_swiftshader.dll", "!**/ffmpeg.dll"] in packaging/electron-builder.directml.yml. Verify on packaged build that nothing breaks.

Touches: packaging/electron-builder.directml.yml · packaging/electron-builder.cpu.yml

11 · SenseVoice in our onnx-asr fork

cost · 2-4 dimpact · closes the only model-availability gap

Axis III. The one family Handy has that we don't. SenseVoice int8 (152 MB, ZH/EN/JA/KO/Cantonese). Port to fork mirroring commit 20b872a (Moonshine + Cohere + Breeze + Canary-180m). After this we have strictly more models than Handy.

Touches: onnx-asr fork (out-of-tree) · server/src/recorder/domain/catalog.json · frontend/packages/model-picker/src/stt/lib/family-helpers.ts

12 · Compile cache for ORT graphs

cost · 4 himpact · ~ 200-500 ms saved per warm load

Axis IV. Set session_options.optimized_model_filepath = cache_dir / f"{model_id}.ort.bin" at session-build time. Reload-from-cache skips graph-optimization on subsequent runs. Invalidate on ORT version bump (__version__ in cache key).

Touches: server/src/recorder/infrastructure/onnxasr_transcriber.py at session-build site

Tier 3 · Defer or skip

What we deliberately don't do — and why.

Touch the Silero CPU pin

Skip. memory/project_silero_vad_cpu_pin_invariant.md is load-bearing — CPU pin is what makes do_copy_in_default_stream=1 safe on CUDA. Moving VAD to GPU re-opens the silent deadlock that bricked the WS dispatcher.

Add denoising / HPF / pre-emphasis to audio path

Skip. Per memory/project_stt_premodel_conditioning_policy.md, denoising raises WER 1-47 %. The far-mic gap closes via prefill + format-priority + resampler quality — all non-destructive changes. The destructive ones are off the table by policy.

Adopt whisper.cpp as a second engine

Defer to post-macOS. The argument is Metal-on-macOS for Whisper. We haven't shipped macOS yet. After macOS lands and CoreML proves insufficient, revisit. Until then: two engines = double test surface for no user-visible win.

Full Tauri migration

Hold. Even at best-case numbers, a migration is a 6-month project that pauses every other feature. We only commit if a scoped prototype shows ≥3× installer reduction and ≥3× cold-start reduction and the eight-window architecture can compose under three WebView engines.

Drop the seed model entirely (vs make on-demand)

Skip. Tier 1 item 6 makes seed-on-demand the default behavior, but the seed remains available for offline-first first transcription on machines that can't reach the internet at install time. That's a feature; keep the code path.

Reduce test rigor "to ship faster"

Skip permanently. Issue №03 named this our deepest moat. None of the Tier 1 actions need any tests reduced; if anything, three of them add new property tests (prefill, unload-idempotent, format-priority).

Exit criteria · when do we know it worked?

  • Far-mic WER ≤ Handy on a controlled bench — VAD prefill + native-SR capture + format-priority shipped.
  • Idle RSS ≤ 300 MB after 5 min idle — model-unload daemon running.
  • Installer ≤ 140 MB compressed after Tier-1 cuts (vs 202 MB today); ≤ 90 MB after Tier-2 runtime-pack.
  • Whisper / Parakeet / Moonshine / Canary fp16 latency ≤ Handy on the same model — ORT_ENABLE_ALL restored on non-Whisper fp16 + intra_op bumped + compile cache shipped.
  • SenseVoice in the catalog — only-remaining-family gap closed.
  • Sentry sample-rate ≤ 0.1 shipped; lazy adm-zip / tar live in main.
  • vk_swiftshader + ffmpeg removed from packaged build; openwakeword optional.
  • Zero features removed. Every shipped capability — diarization, wake-word, cloud STT, translate, context-awareness, custom modifiers, listen mode, TTS pack, history, dictionary — still works post-migration.
