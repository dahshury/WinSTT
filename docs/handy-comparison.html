Stack: Electron · Python · ONNX
An Action Plan, not a Comparison

What to
borrow from
Handy.

Sixteen features Handy ships that we don't — paired with the exact path to land each one on our stack. We're not switching to Tauri. We're not dropping ONNX. We're taking what works and bolting it onto what we have.

Non-negotiable · 01

Electron stays.

The Electron + Next.js shell is the spine. We're not rewriting the renderer in WebKit/Tauri. Implication: every adopted UX feature has to be reachable from electron/ main-process APIs, native Node addons, or shell-outs.

Non-negotiable · 02

ONNX stays.

Our entire model catalog — Distil-Whisper, Lite-Whisper, Moonshine, Cohere, Parakeet, Canary — is ONNX. Implication: no whisper.cpp swap. But: Handy's transcribe-rs already runs ONNX alongside whisper.cpp, which is academic — what we actually need from them is the EP strategy per OS (DirectML, CoreML, OpenVINO), all of which we already have ports for.

Open · 03

Everything else.

Native Node addons, child-process shell-outs, Python pipeline changes, new npm packages, schema changes, SQLite, native menu bridges — all on the table. The job: map every Handy strength to a concrete change inside our existing stack.

Tier A · Adopt this quarter
01PASTE

Six paste methods + auto-submit + trailing-space

Equivalent
A configurable PasteMethod enum (CtrlV / CtrlShiftV / ShiftInsert / Direct-type / None / ExternalScript), paste_delay_ms, append_trailing_space, auto_submit with Enter/Ctrl-Enter/Cmd-Enter, and on Linux a five-tool fallback chain through wtype/kwtype/dotool/ydotool/xdotool.

This is our single highest-leverage borrow — and it's pure Electron work. None of our Python pipeline is touched.

  1. Replace globalShortcut-based re-paste with a dispatcher in electron/ipc/relay.ts. New enum PasteMethod in spec/openapi.yaml; routed via existing IPC.
  2. Add @nut-tree-fork/nut-js (~30 MB native, cross-platform) for typing-simulation. Wraps the same OS primitives Handy reaches for. Direct mode = keyboard.type(); CtrlV = keyboard.pressKey() sequence with platform-virtual-keys; ShiftInsert = same.
  3. Auto-submit is one IPC + one settings boolean. After paste resolves, dispatch the configured submit key via the same nut.js path.
  4. ExternalScript is child_process.execFile(script, [text]) from main. Forty lines including timeout + stderr capture. Power-user feature parity.
  5. On Linux, mirror Handy's clipboard.rs tool detection verbatim: shell out to which wtype / kwtype / dotool / ydotool / xdotool, prefer based on Wayland-vs-X11 + KDE detection. Falls back to nut.js for portability.

Memory we already have around this: "Re-paste = only Electron globalShortcut" — this redesign explicitly retires that limitation.

Touches Electron main Renderer (settings UI) nut.js addon spec/openapi.yaml
02CLI

CLI flags + single-instance forwarding + Unix signals

Copy
A clap-defined CLI (--toggle-transcription, --start-hidden, --cancel…). The second invocation forwards argv through tauri-plugin-single-instance to the running process, then exits. On Unix, SIGUSR1/SIGUSR2 map to the same actions.

Direct copy. Electron has every piece of this natively.

  1. Single-instance lock in electron/main.ts via app.requestSingleInstanceLock(). The second-instance event already passes argv to the first instance.
  2. Argv parsing with commander or plain code — we don't need clap-grade ergonomics. Flags map 1:1 to existing IPC commands.
  3. Unix signals: process.on('SIGUSR2', toggleTranscribe) in main. Six lines. Node refuses SIGUSR on Windows — wrap in if (process.platform !== 'win32').
  4. Document the WM-integration pattern in docs/: bind Sway/Hyprland/KDE custom shortcut to winstt --toggle-transcription or to pkill -USR2 -n winstt. Wayland users get hotkey support without us solving Wayland.

This is the unlock for Linux as a real platform — without it, Wayland users can't bind global hotkeys at all.

Touches electron/main.ts scripts/dev-electron.ts
03PORT

Portable mode

Copy
If a file literally named portable sits next to the exe, every piece of user data (settings, models, recordings, db, logs) goes to ./Data/ instead of %APPDATA%.

Eighty lines in Handy's portable.rs. Roughly the same in our TypeScript main.

  1. Detection: at boot, before any path resolution, check for path.join(path.dirname(app.getPath('exe')), 'portable').
  2. Path override: app.setPath('userData', dataDir) + app.setPath('logs', dataDir/logs) + app.setPath('temp', …). Done before any module reads paths.
  3. electron-store already inherits from app.getPath('userData') — automatic.
  4. Python server: pass --data-dir arg from main process spawning the server. Server resolves models, recordings, logs relative to it.
  5. electron-builder.yml: add a third NSIS target with "portable": true that emits a single-file portable exe. The marker file is created on first run if missing.

Huge value for IT-locked corporate users, USB sticks, and the "I don't want this in AppData" crowd. Eighty lines is one afternoon.

Touches electron/main.ts server/bootstrap.py electron-builder.yml
04HIST

SQLite history + WAV retention + tray submenu

Copy
A real history table (file_name, timestamp, saved, title, raw text, post-processed text, prompt). WAV files saved concurrently with transcription and verified. recording_retention_period garbage-collects on a schedule. Tray submenu shows the last N entries with click-to-copy.

This is a feature gap regardless of cross-platform. We already keep "last transcription" — extending it to a full table is incremental.

  1. Add better-sqlite3 in the Electron main process (we already use @electron/rebuild for native modules; one more line in rebuild script). Sync API is fine on a background thread.
  2. Schema mirrors Handy's: transcription_history(id, file_name, ts, saved, title, text, post_processed_text, post_process_prompt, post_process_requested). Migrations as a sorted-array of SQL strings, tracked via PRAGMA user_version.
  3. WAV persistence: Python server already produces samples — add an optional save_wav: bool in the recorder's done-event, write to {dataDir}/recordings/handy-{ts}.wav in the Python side, emit the filename back to main, save the DB row.
  4. Retention: a 1-hour setInterval in main process. Deletes rows + WAV files older than the configured period.
  5. Tray submenu: build dynamically from a history.recent(10) query on every tray-icon click. Click-to-copy → clipboard.writeText().
  6. UI page at /history in the renderer. FSD slice entities/transcription-history + a views/HistoryView.
Touches electron/ipc/history.ts (new) better-sqlite3 recorder save-WAV hook new views slice
05DICT

Local fuzzy custom-dictionary (Levenshtein + Soundex + n-gram)

Copy
~200 lines of Rust in audio_toolkit/text.rs: lowercase + Levenshtein + Soundex phonetic match + n-gram for multi-word artifacts (e.g. "Charge B" → "ChargeBee"). Length-percentage early-exit. Threshold configurable.

Port to Python so it runs inside the existing transcription pipeline — zero IPC round-trip. Today our dictionary correction is LLM-driven (modifier pipeline), which costs tokens; this is free and offline.

  1. Add jellyfish (Soundex) + rapidfuzz (fast Levenshtein) to server/pyproject.toml. Both pure-Python or ship wheels for every OS — no platform-specific compilation.
  2. Translate apply_custom_words() 1:1. Same algorithm: build n-grams, score by Levenshtein normalized by length, boost phonetic matches by 0.3x, accept under threshold.
  3. Drop into the post-transcription hook in RecorderService, before the modifier pipeline. If the dictionary fixes "ChargeBee", the LLM never has to see it.
  4. Settings UI: customWords: string[] + wordCorrectionThreshold: number. Already roughly in our settings schema — extend, don't add.

This is purely additive. Keep the LLM-modifier path as the heavy-duty fallback; the fuzzy matcher handles the 80% case for free.

Touches server/src/recorder/text/ post-transcription hook
06LLM

Multi-provider LLM post-processing

Equivalent
Eight default providers (OpenAI, Anthropic, Z.AI, OpenRouter, Groq, Cerebras, AWS Bedrock, Custom→Ollama) plus Apple Intelligence on macOS ARM. Per-provider supports_structured_output with JSON-schema enforcement. Strips invisible Unicode. Prompt library with named prompts.

We already have the AI SDK on the Electron main side for cloud STT. Adding LLM post-processing reuses the same provider abstraction.

  1. Provider registry in electron/ipc/llm-providers.ts: same schema as Handy's PostProcessProvider. Read from settings; default-merge on load.
  2. AI SDK already abstracts @ai-sdk/openai, @ai-sdk/anthropic, @ai-sdk/groq, etc. Per-provider auth quirks (Anthropic x-api-key) are handled by the SDK. Structured outputs via generateObject({ schema }).
  3. Redacted-log primitive: copy Handy's SecretMap idea — a thin TS wrapper whose toJSON returns [REDACTED] for non-empty values. Used everywhere we log settings.
  4. Ollama-as-Custom: the Custom provider defaults to http://localhost:11434/v1. AI SDK has a @ai-sdk/openai-compatible exactly for this.
  5. Apple Intelligence: defer to Tier C. Path is clear (Swift CLI bundled, called via child_process) but only valuable post-macOS-launch.
  6. Our modifier UI keeps its identity. The provider/prompt layer is the engine; modifier toggles are the curated UX on top. They compose.
Touches electron/ipc/llm-*.ts AI SDK providers spec/openapi.yaml
07TRAY

State-driven tray icon + history submenu + theme variants

Copy
Three states (Idle / Recording / Transcribing), three theme palettes (dark / light / Linux-colored). Tray menu shows the last N transcriptions with click-to-copy.

Electron's Tray API has setImage() and setContextMenu(). Rebuilding the menu on every state change is fine.

  1. Asset prep: produce tray_{idle,recording,transcribing}_{dark,light,color}.png @ 2x. Pull from existing brand identity in docs/identity/.
  2. State machine in electron/ipc/tray.ts already exists — extend to subscribe to recorder events and call tray.setImage() on transitions.
  3. Theme detection: nativeTheme.shouldUseDarkColors + nativeTheme.on('updated', …) picks dark/light. Linux always gets colored.
  4. History submenu piggybacks on #04. Menu.buildFromTemplate, click handler → clipboard.writeText.
Touches electron/ipc/tray.ts assets pipeline
08DL

Cancellable downloads + SHA-256 + auto-extract

Copy
Per-model AtomicBool cancel flag, SHA-256 verification on download, automatic .tar.gz extraction for Parakeet models into named directories. DownloadCleanup RAII guard ensures state always resets.

Pure Electron-main + Python. We already do downloads, just without these guarantees.

  1. Cancellation: AbortController per active download, stored in a Map<modelId, AbortController>. UI cancel button calls abort().
  2. SHA-256: Node crypto.createHash('sha256') hooked into the stream. Compare to manifest. Manifest lives in spec/openapi.yaml or alongside the model catalog.
  3. Tar extraction: tar npm package, ~30 lines. Already needed for our offline TTS pack pattern.
  4. Cleanup guarantee: TypeScript doesn't have RAII, but a try/finally in the download orchestrator gives the same property.
  5. Progress events: we already have download-progress UI from the recent updater work. Reuse the event shape.
Touches electron model-download IPC model-catalog store
Tier B · Adopt next quarter
09EP

Configurable GPU per backend + DirectML shrink

Equivalent
User-selectable whisper_accelerator (Auto/CPU/GPU) and ort_accelerator (Auto/CPU/CUDA/DirectML/ROCm). Whisper-cpp gets Metal on mac, Vulkan on Win/Linux. ONNX gets DirectML on Win, CUDA on Linux. Their GPU build is ~200 MB. Ours is 2 GB.

The 2 GB GPU bundle is our biggest distribution problem. The fix doesn't require changing ML stacks — it requires changing which ONNX execution provider we ship. Comments in server/src/recorder/infrastructure/device.py already reference DirectML and ROCm; we never finished the work.

  1. Add a [directml] extra in server/pyproject.toml with onnxruntime-directml. ~200 MB; works on AMD / Intel / NVIDIA via DirectX 12. Drops the eight NVIDIA CUDA wheels entirely on this build.
  2. Add a [coreml] extra with onnxruntime-silicon (or stock onnxruntime with CoreML EP). Native on Apple Silicon, no extra wheels.
  3. Add an [openvino] extra for Linux Intel iGPU acceleration where CUDA isn't available.
  4. Extend device.py from CUDA-or-CPU to a priority-ordered list: try EP in order, fall through to CPU. Honor a user setting accelerator: "auto" | "cuda" | "directml" | "coreml" | "openvino" | "cpu".
  5. Restructure release artifacts: WinSTT-Win-DirectML-Setup.exe (200 MB, default), WinSTT-Win-CUDA-Setup.exe (2 GB, power users), WinSTT-macOS-Setup.dmg (CoreML), WinSTT-Linux.AppImage (CPU + optional OpenVINO).

Result: same ONNX stack, same model catalog, much smaller default download, GPU on Intel/AMD Windows users for the first time.

Touches onnxruntime EP selection device.py packaging/electron-builder.*.yml
10KEYS

Two hotkey backends with auto-fallback

Equivalent
Try handy-keys first; on init failure fall back to tauri-plugin-global-shortcut and persist the choice in settings. Per-named-binding (transcribe / post-process / cancel) independently configurable. Cancel is dynamically registered only during recording.

We have uiohook-napi (passive listener) + Electron globalShortcut (active grab). They have different failure modes. Build the same try-and-persist pattern.

  1. Setting: keyboardImplementation: "uiohook" | "globalShortcut", default "uiohook" on Win/macOS, "globalShortcut" on Linux.
  2. Init wrapper in electron/ipc/hotkey.ts: try primary, catch init failures, swap to secondary, store.set('keyboardImplementation', secondary) so the fallback persists.
  3. Per-named-binding: extend the settings schema from a single-hotkey-per-mode to bindings: Record<BindingId, ShortcutBinding>. Migrate existing values on load.
  4. Dynamic cancel: register Escape only when the recorder enters Recording state; unregister on Idle. Stops accidentally trapping Escape in apps.
Touches electron/ipc/hotkey.ts settings schema migration
11i18n

i18next with locale support + ESLint enforcement

Copy
Twenty locales, i18next, eslint-plugin-i18next rule that fails CI if a JSX literal isn't wrapped in t(). Tray menu localizes via tray_i18n.rs. OpenCC converts Simplified ⇄ Traditional Chinese after transcription. RTL helpers.

Our Next.js app supports next-intl natively. Use that for the renderer; use i18next + a small JSON loader in main for tray strings.

  1. Add next-intl to frontend/, restructure src/shared/i18n/ with one JSON per locale. Start with en; ship others via community contribution.
  2. ESLint rule: configure Biome's noJsxLiterals or layer ESLint with eslint-plugin-i18next in frontend/biome.json ignore + eslint config. CI fails on bare strings.
  3. Tray strings: tiny electron/i18n/ with { locale: { trayIdle, trayRecording, … } }. Locale comes from settings.
  4. OpenCC for zh-Hans ⇄ zh-Hant: there's a Python port (opencc-python) — drop it into the same post-transcription hook as the dictionary.
  5. RTL: Next.js dir="rtl" attribute + Base UI's RTL support. Mostly free.

The big cost is the sweep: every JSX literal in frontend/src/ has to migrate. One-time pain, sustained by the ESLint guard.

Touches frontend (full sweep) electron/i18n/ post-transcription (OpenCC)
12CMOD

Custom user-model auto-discovery

Copy
Drop a Whisper GGML .bin in models/, restart, it appears under "Custom Models" with the filename as the display name.

Equivalent for our ONNX world: drop a HuggingFace-style ONNX folder (encoder + decoder + config) and we discover it.

13SIG

Minisign-signed releases

Equivalent
Tauri's updater uses minisign signatures, with the pubkey embedded in tauri.conf.json. README walks users through manual verification with the minisign CLI.

We already use electron-updater, which has its own signing flow (Authenticode on Windows, codesign on macOS). Adding minisign on top is a power-user trust layer.

  1. Generate a minisign keypair, commit pubkey to repo, keep secret key as a GitHub Actions secret.
  2. CI step in .github/workflows/electron-release.yml: after build, sign every installer artifact with minisign; upload .sig alongside.
  3. Document verification in docs/ with the same recipe Handy publishes.
  4. Optional: validate signature in the updater path itself before launching the new version. Lower priority — electron-updater already validates code signatures.

14HUD

Per-OS overlay primitives (NSPanel + layer-shell)

Custom
macOS uses tauri-nspanel — a true floating panel that doesn't steal focus from the app the user wants to type into. Linux uses gtk-layer-shell for compositor-level layering, with an HANDY_NO_GTK_LAYER_SHELL=1 escape hatch. Linux defaults to no overlay because some compositors break paste with it on.

Electron's BrowserWindow can almost match NSPanel via setAlwaysOnTop('screen-saver', 1) + setVisibleOnAllWorkspaces({ visibleOnFullScreen: true }) + setFocusable(false). It's not as clean — focus management is a known Electron sore spot on macOS — but it ships.

  1. macOS path A (cheap): use the above BrowserWindow flags. Test against Slack / VSCode / browsers to confirm paste lands in the previous focused app. If it does, ship.
  2. macOS path B (expensive): write a small N-API native addon that wraps NSPanel via NodeRT-style bridging. ~300 lines of Obj-C++ + TypeScript wrapper. Probably defer to v1.1.
  3. Linux: no off-the-shelf gtk-layer-shell binding for Node. Default overlay to none on Linux (same default Handy ships). If a user wants it, they get a regular always-on-top window and the documented caveats.
  4. Windows: already done. Keep the existing implementation.

Honest read: path A is enough for v1.0 macOS. Most users won't notice the difference.

Touches overlay BrowserWindow flags (optional) NSPanel addon
15AI

Apple Intelligence as a local LLM provider

Custom
Native Swift FFI to Apple's FoundationModels framework on macOS ARM. No API key. Fully on-device. Used as a post-process provider. Availability check deferred to first use (early init crashes the app on macOS 26 beta).

There's no Node binding for FoundationModels. The clean path is to bundle a tiny Swift CLI inside the macOS app and call it from Electron main.

  1. Tiny Swift CLI (~80 lines): takes stdin JSON { system, user }, calls SystemLanguageModel.default, prints stdout JSON. Compiled per-architecture.
  2. Bundle in resources/macos/winstt-apple-llm via electron-builder's extraResources. Code-sign as part of the regular macOS signing pass.
  3. Provider adapter in electron/ipc/llm-providers.ts: if provider === "apple_intelligence" and process.platform === "darwin", shell out to the CLI. Availability check on first call (avoid Handy's early-init crash issue).
  4. Settings UI hides the provider on non-Apple-Silicon platforms.

Realistic but a v1.1 feature. Cloud-API providers cover the same use case for v1.0.

Touches Swift CLI (new) LLM provider routing macOS code-signing
16LID

Clamshell-mic switching

Custom
Detects laptop lid close and swaps to a configured alternate microphone (e.g. external USB mic when the built-in disappears).

Niche but charming. Detection is OS-specific.

  1. macOS: IOPMrootDomain notifications. Tiny native addon or caffeinate-style polling of pmset -g.
  2. Linux: poll /proc/acpi/button/lid/LID*/state at 1 Hz. Twenty lines.
  3. Windows: WM_POWERBROADCAST via a native message hook. Or skip — clamshell-mode is far less common on Windows laptops.
  4. Settings: clamshellMicrophone: string | null. Empty = no swap.

Pure UX nice-to-have. Defer past the cross-platform launch.

Touches main process lid watcher macOS addon (optional)

How WinSTT reaches three platforms

With Electron and ONNX preserved, the cross-platform job decomposes into three workstreams: Python server portability, Electron main-process platform-branching, and per-OS feature scope. Here's the lay of the land.

Windows

Status · Shipping
Server bundling
PyInstaller → stt-server.exe. Already in packaging/. Two flavors today.
ONNX EP
Add DirectML. Drops installer to ~200 MB and unlocks AMD / Intel GPUs.
Hotkeys
uiohook-napi + Electron globalShortcut. Both work. Status quo.
Paste
Existing path stays. Becomes one of six methods.
Overlay
BrowserWindow always-on-top. Status quo.
Signing
Authenticode via electron-builder. Add minisign sidecars.
Wakeword + Loopback
Stays. WASAPI is Win-only and remains a flagship Windows feature.

macOS

CoreML EP via onnxruntime-silicon. Native Apple Silicon path. CPU fallback on Intel.

Linux

Target · v1.1
Server bundling
PyInstaller → ELF. AppImage for the Electron app. Test on Ubuntu 22.04 / 24.04 + Arch.

CPU baseline; OpenVINO EP for Intel iGPU; CUDA EP for NVIDIA users (drop the GPU bundling tax — let users install via system package manager).
Hotkeys

The Three-Phase Plan

iFoundation

Adopt the Tier-A wins on Windows

Ships on Windows · v0.9
  • Six paste methods (#01), CLI flags + signals (#02), portable mode (#03)
  • SQLite history (#04), local fuzzy dictionary (#05)
  • Multi-provider LLM via AI SDK (#06)
  • State-driven tray + theme variants (#07), cancellable downloads (#08)
  • Add DirectML EP as the default Windows GPU build (#09). 2 GB CUDA build becomes the "advanced" option.

Net effect: Windows version becomes dramatically better — and the codebase is now cross-platform-ready in all the renderer/main-process places. No Python platform-porting yet.

iiApple

macOS support, feature-degraded

Ships on Windows + macOS · v1.0
  • PyInstaller server build on macOS, Developer ID signing + notarization for both binaries
  • CoreML EP wired into device.py (#09)
  • Hotkey fallback chain (#10) + onboarding for Accessibility permission
  • Overlay BrowserWindow with NSPanel-imitation flags (#14)
  • i18next sweep complete (#11) — community contributes locales
  • Whisper custom-model auto-discovery (#12), minisign sidecar (#13)
  • Windows-only features: loopback "listen mode", UIA-based context-awareness. Documented as such.
iiiPenguin

Linux support, Wayland-aware

Ships on all three · v1.1
  • PyInstaller ELF + AppImage packaging
  • Linux paste fallback chain (built in phase i, validated here)
  • Wayland strategy: CLI flags + Unix signals are the integration surface (built in phase i)
  • OpenVINO EP for Intel iGPUs (#09)
  • AppIndicator detection + degradation
  • Apple Intelligence as LLM provider (#15) — macOS bonus
  • Optional: clamshell-mic switching (#16), NSPanel native addon
Skipping · Not worth adopting

Whisper.cpp instead of ONNX

Handy uses whisper-rs for Whisper and transcribe-rs ONNX for everything else. We use ONNX for both. They get Metal/Vulkan via whisper.cpp; we get the same OS coverage via DirectML/CoreML EPs of ONNX. Switching ML stacks costs us Lite-Whisper, Distil-Whisper, Cohere, and our entire model-export tooling. The EP-swap path is strictly cheaper.

Tauri rewrite

Cleaner end state, but a multi-month rewrite that forces us to either FFI into Python (slow) or port the recorder to Rust (huge). Our Electron + Python topology already reaches three platforms — the work is incremental, not structural.

Drop wakeword + loopback for parity

Two of our most distinctive features. They're Windows-only and that's fine. Cross-platform parity isn't more valuable than feature differentiation.

Replace modifier pipeline with free-form prompts

Handy's prompt library is more general; ours is more opinionated. Keep the curated modifier UX as our identity. The provider/structured-output engine borrowed in #06 sits underneath the modifier UI without replacing it.
