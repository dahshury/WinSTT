When something goes sideways, WinSTT's job is to make it obvious what happened. Three tools cover almost every case: the `--debug` CLI flag, the rolling `debug.log` / `stt-server.log` files, and the one-click **Save Diagnostic Bundle**.

## Quick tour

| Tool | What it does | Where to find it |
|---|---|---|
| `--debug` flag | Bumps terminal log level to verbose for the current launch. The file log is always verbose, so this is just for live tailing. | Pass when launching `WinSTT.exe` or `uv run stt-server`. |
| `debug.log` | Electron main + renderer logs (5 MB rolling, rotates to `debug.old.log`). | `%APPDATA%/WinSTT/debug.log` |
| `stt-server.log` | Python STT server logs. Same rotation. | `%APPDATA%/WinSTT/stt-server.log` |
| **Save Diagnostic Bundle** | Zips both logs + GPU info + redacted settings into one file. | Settings → Advanced → **Diagnostics** → **Save Diagnostic Bundle…** |

## Generating a diagnostic bundle

The fastest path to a bug fix is attaching the diagnostic bundle to your issue. The bundle is **redacted** — settings keys flagged as secrets (API keys, OAuth tokens) are scrubbed before the zip is written.

<Steps>

<Step>
### Reproduce the issue with `--debug`

Quit WinSTT, then launch from a shortcut or terminal with `--debug` appended:

<Tabs items={['PowerShell', 'cmd.exe', 'Run dialog']}>
<Tab value="PowerShell">
```powershell
& "$env:LOCALAPPDATA\Programs\WinSTT\WinSTT.exe" --debug
```
</Tab>
<Tab value="cmd.exe">
```bat
"%LOCALAPPDATA%\Programs\WinSTT\WinSTT.exe" --debug
```
</Tab>
<Tab value="Run dialog">
Press `Win+R` and enter:

```
"%LOCALAPPDATA%\Programs\WinSTT\WinSTT.exe" --debug
```
</Tab>
</Tabs>

The file log is always verbose, so even if you forget the flag, `debug.log` and `stt-server.log` still contain the full trace. The flag matters mainly when you want to tail the terminal yourself.
</Step>

<Step>
### Reproduce, then save the bundle

Do whatever caused the bug — short enough that the relevant log lines are recent (`debug.log` rolls at 5 MB).

Then:

1. Open **Settings**.
2. Scroll to **Advanced**.
3. Find the **Diagnostics** section.
4. Click **Save Diagnostic Bundle…** and pick a target path.

The zip contains:

```
diag-bundle-<timestamp>.zip
├── debug.log
├── debug.old.log         (if a rotation happened recently)
├── stt-server.log
├── stt-server.old.log
├── gpu-info.txt          (dxdiag-style output + ONNX Runtime EPs probed)
├── settings.redacted.json
└── meta.json             (WinSTT version, OS, CPU, screen DPI, etc.)
```
</Step>

<Step>
### Attach the bundle to your bug report

Drop the `.zip` into the GitHub issue. The default text of the issue template (`.github/ISSUE_TEMPLATE/bug_report.md`) asks for it explicitly.

If the file is over GitHub's 25 MB attachment limit, attach `meta.json` + the tail of each log (`Get-Content -Tail 500 debug.log`) inline and upload the full bundle to a [gist](https://gist.github.com) or a cloud drive link.
</Step>

</Steps>

## Reading the logs yourself

Both log files use the format `[HH:MM:SS.mmm] [scope] message`. Useful scopes:

| Scope | Source |
|---|---|
| `stt-process` | Spawn / restart of the bundled `stt-server.exe`. Any "exit code 1" early in the file is the cause of the windowed crash. |
| `ws-client` | Electron's WebSocket client to the server. Reconnects, disconnect storms (see `memory/project_ws_reconnect_storm.md`). |
| `relay` | IPC relay between Electron main and the renderer. |
| `settings` | Settings persistence. Look here when settings appear to revert. |
| `hotkey` | Global hotkey registration. `globalShortcut.register false` means another app claimed the chord. |
| `diag-bundle` | The diagnostic-bundle generator itself. |

Server side (`stt-server.log`):

| Scope | Source |
|---|---|
| `bootstrap` | Adapter wiring at startup. The first 50 lines say which audio device, VAD, transcriber, and wake-word backend were resolved. |
| `recorder.application` | `RecorderService` lifecycle, state-machine transitions. |
| `recorder.pipeline` | VAD loop, per-chunk decisions, turn boundaries. Very chatty in `--debug`. |
| `model_swap` | Hot-swap of STT models. Failures here usually mean an incompatible quantization for the chosen backend. |

## Sentry crash reports

If a crash gets through, the Electron main process forwards a redacted exception payload to Sentry. This is **opt-out** and can be disabled at **Settings → General → Anonymous crash reports**. The redactor strips file paths, the home directory, and anything that pattern-matches an API key. See `frontend/SENTRY.md` for the full schema.

When you file a bug, Sentry isn't a substitute for the diagnostic bundle — Sentry sees the *crash*, not the *behavior leading up to it*. The bundle has the latter.

## What "verbose" actually logs

To answer the most common question on bug reports — **no, the verbose log does not contain your audio**. Audio frames are processed in memory and discarded; only summary statistics (chunk count, VAD decision, transcribed text length) make it to the log. Transcribed *text* is logged at debug level so you can correlate "what I said" with the model's output, but the redactor in the diagnostic bundle masks anything that looks like an email, phone number, or credit card before zipping.

If you'd rather not have transcribed text in the log even at debug level, set the env var `WINSTT_LOG_NO_TEXT=1` before launch. The bundle generator respects this and writes `<redacted by WINSTT_LOG_NO_TEXT>` in place of every transcription.
