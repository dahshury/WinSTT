title: Frontend Architecture
description: Electron + Vite multi-page React with Feature-Sliced Design

The WinSTT frontend is an Electron application with a Vite multi-page React 19 renderer. Each Electron BrowserWindow loads its own HTML entry directly via `file://` -- there is no router and no dev server in production. All server communication happens in the Electron main process; the renderer has zero WebSocket code.

## Process Architecture

### Electron Main Process

The main process handles:

- **WebSocket connection** to the Python STT server (dual channels: control JSON + binary audio)
- **Python process management** -- spawning and killing the STT server
- **System tray** integration and window management
- **Global hotkey** registration and event forwarding
- **IPC relay** between the WebSocket connection and the renderer

Key files:

- `electron/main.ts` -- Entry point, window creation, process lifecycle
- `electron/ws/stt-client.ts` -- Dual-channel WebSocket client
- `electron/ipc/*.ts` -- Modular IPC handlers, grouped by concern:
  - **Server / relay** -- `relay.ts` (dual-channel WebSocket relay), `stt-process.ts` (Python subprocess lifecycle), `stt-commands.ts` (parameter get/set, model reload)
  - **Hotkeys** -- `hotkey.ts` (push-to-talk uiohook listener), `transform-hotkeys.ts` (custom-transform hotkeys), `tts-hotkey.ts` (speak-selection hotkey)
  - **LLM / transforms / TTS** -- `llm.ts` (Ollama scan, OpenRouter, dictation/transform text processing, warm-up), `transforms.ts` (custom LLM transforms on selected text), `tts.ts` (Kokoro TTS synthesis)
  - **Files & overlay** -- `file-transcribe.ts` (file picker, upload, progress, txt/srt), `overlay.ts` (floating recording overlay window)
  - **Persistence & diagnostics** -- `transcription-history.ts` (history persist + clear), `diag-bundle.ts` (logs/config/system info bundle), `settings.ts` (schema, startup migration, secret encryption)
- `electron/lib/context-reader.ts` -- Reads focused-window text via Windows UI Automation for the opt-in Context Awareness feature

### Preload Bridge

`electron/preload.ts` exposes IPC channels to the renderer via `contextBridge`. It strips `IpcRendererEvent` from callbacks so renderer code receives clean arguments.

### Renderer

The renderer is a Vite 7 multi-page app. Each Electron BrowserWindow loads its own HTML entry from `windows/<page>.html` (with `index.html` for the main window). One React tree per entry under `src/entries/<page>.tsx`. It uses Feature-Sliced Design for code organization.

## Feature-Sliced Design Layers

```
src/
  app/        Bootstrap, providers, global styles
  views/      One slice per BrowserWindow (main, settings, overlay, tray-menu,
              model-picker, device-picker, onboarding, history)
  widgets/    Composite blocks combining features and entities
  features/   Reusable user interactions
  entities/   Business domain objects
  shared/     Foundation utilities, UI primitives, config
  entries/    One .tsx per HTML window (createRoot + render <View />)
```

### Import Contract

Each layer may only import from layers below it. Never sideways.

| Layer | May Import From |
|-------|----------------|
| app | all layers |
| views | widgets, features, entities, shared |
| widgets | features, entities, shared |
| features | entities, shared |
| entities | shared |
| shared | nothing |

## State Management

| Tool | Purpose |
|------|---------|
| Zustand | Renderer client state (settings, model catalog, transcription) |
| IPC | Server-of-truth data layer (no HTTP, no TanStack Query, no Redux) |
