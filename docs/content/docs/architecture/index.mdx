WinSTT is a **single-process desktop application** built with [Tauri 2](https://tauri.app):
a **Rust backend** and a **WebView2-rendered React frontend**, compiled into one native
executable. There is no separate server and no network hop — speech recognition and
text-to-speech run **in-process in Rust** on [ONNX Runtime](https://onnxruntime.ai) (`ort`).

## The two layers

<BentoGrid cols={2}>
  <BentoCell icon="cpu" title="Backend — Rust" href="/docs/architecture/backend">
    `src-tauri/`. Owns audio capture, voice-activity detection, STT/TTS model loading and
    inference, the LLM/cloud integrations, settings, history, and the global hotkey.
    Organized under the `winstt::*` module tree.
  </BentoCell>
  <BentoCell icon="waveform" title="Frontend — React 19" href="/docs/architecture/frontend">
    Vite multi-page React 19, Feature-Sliced Design (`src/`). It renders the recording pill,
    settings, model picker, history, and overlays across several Tauri windows.
  </BentoCell>
</BentoGrid>

The renderer holds no model code and opens no sockets. It reaches Rust two ways, both
type-checked end-to-end by [`tauri-specta`](https://github.com/specta-rs/tauri-specta).

<StepFlow>
  <Step title="Tauri commands — typed request/response">
    The React renderer calls into Rust through `#[tauri::command]` functions: a typed
    request goes out, a typed response comes back.
  </Step>
  <Step title="Events — push notifications">
    Rust pushes updates the other direction as events, so the UI reflects recording
    lifecycle, download progress, and other state without polling.
  </Step>
</StepFlow>

<Callout type="info" title="One binary, no sockets">
  WinSTT is a single binary: STT and TTS run in-process in Rust on `ort`, and the renderer
  reaches the backend through Tauri commands + events rather than any network relay. A thin
  native-bridge adapter maps the renderer's IPC calls onto Tauri.
</Callout>
