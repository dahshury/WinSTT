`stt-server` is the Python speech engine that does the actual transcription; the Electron app spawns it and talks to it over two local WebSockets. You can also launch it by hand — for a headless box, CI, or to debug a setting in isolation.

```bash
stt-server [OPTIONS]
```

In a packaged install the same binary ships as `stt-server.exe` under the app's resources. From a source checkout, run it through the `server/` virtualenv:

```bash
# from the repo root
uv run --project server stt-server -m tiny --debug
```

## How it runs

The server opens two WebSockets and waits for the Electron app (or any client) to connect: a **control** channel for commands and a **data** channel for audio + live transcription. The Electron app sets almost every flag for you from the settings UI — you only need these directly when running `stt-server` standalone.

<Callout type="info" title="Most flags accept two spellings">
  Flags are registered with both a dashed and an underscored alias (for example `--rt-model` / `--realtime_model_type`, `--log-dir` / `--log_dir`). Either works; the tables below list the canonical form first.
</Callout>

<Callout type="warn" title="Only the model is persisted">
  `stt-server` writes a `server-settings.json` next to your data dir, but it only persists the **model** choice. Every other flag is a per-launch value — pass it again next time. On startup, a saved model is used unless you override it with `-m` / `--model`.
</Callout>

A typical standalone launch — small model on CPU, default microphone, verbose logs:

```bash
stt-server -m tiny --device cpu -i 1 --debug
```

Force a non-default model + quantization on DirectML:

```bash
stt-server -m large-v3-turbo --device directml --onnx_quantization fp16
```

  It only does anything when `--model` is a multilingual Whisper export. It's silently ignored on `.en` variants and on NeMo, GigaAM, and Moonshine families.

## History recordings

The server can persist each transcribed utterance's audio so the history UI can play it back.

<ModelTable
  head={["Flag", "Default", "What it does"]}
  rows={[
    ["--save_wav, --save-wav", "off", "Persist each utterance's PCM to a WAV and report its path on the fullSentence event."],
    ["--recordings_dir, --recordings-dir", "(empty)", "Directory the server writes history WAVs into. Empty disables WAV writing even with --save_wav set."],
  ]}
  numeric={[1]}
/>

<Callout type="info" title="Both are required together">
  `--save_wav` writes nothing unless `--recordings_dir` is also set. The Electron app passes both, pointing the directory at `%APPDATA%/winstt/recordings` so saved files line up with the history store.
</Callout>

  When the Electron app runs in portable mode it passes `--data-dir` (and exports `WINSTT_DATA_DIR`) so every server-side artifact — `server-settings.json`, models, the log, recordings — lives under one tree instead of `~/.winstt` and the platform cache.
</Callout>
